# LP-01 Daily Status Loop

LP-01 - ежедневный schedule loop для PMO-контроля проектов. Сценарий собирает статусы от РП, нормализует их в RAG, выявляет аномалии, запускает эскалации и формирует ежедневный dashboard.

## Назначение

Сценарий нужен, чтобы к 14:00 получить управленческую картину по активным проектам:

- какие проекты Green/Yellow/Red;
- какие вехи под угрозой;
- где нет актуального статуса;
- какие аномалии требуют эскалации;
- какие письма/уведомления нужно отправить или согласовать;
- что уже записано в dashboard, outbox и журнал.

В сценарии используются только:

- `Скилы/external/SK-01-project-status`
- `Скилы/external/SK-06-escalation`
- `Скилы/external/LP-01-daily-pmo-control`

## Как работает цикл

Нормативный рабочий день LP-01:

| Время | Действие |
|---|---|
| 09:00 | Запуск `SK-01`. Агент проверяет активные проекты и запрашивает статусы у РП / ответственных. |
| До 12:00 | Контроль покрытия: нужно получить статусы минимум по 95% проектов. Статус считается актуальным, если обновлён меньше 24 часов назад. |
| 13:00 | Follow-up: если нет ответа 4 часа, агент готовит напоминания и кандидатов на эскалацию. |
| До 14:00 | Агент рассчитывает RAG, анализирует аномалии и передаёт их в `SK-06`. |
| 14:00 | Агент формирует dashboard, escalation check, outbox и loop log. Dashboard направляется агенту стрима и руководителю стрима. |

Полный цикл должен занимать не больше 5 часов.

## Режимы работы

### Тестовый режим

Тестовый режим используется для проверки логики на синтетических XLSX из папки `Входные данные/`.

В тестовом режиме агент:

1. Читает тестовые таблицы.
2. Сам рассчитывает RAG, аномалии и цепочки эскалаций.
3. Не использует заранее подготовленные expected-результаты.
4. Может использовать сжатые временные пороги в минутах вместо реальных 4ч/8ч/24ч.
5. Всегда начинает тестовые цепочки с L1, чтобы проверить матрицу, переходы и остановку по ответам.
6. Формирует planned outbox rows.
7. Записывает выходные XLSX, если пользователь просит или разрешает запись.
8. Не отправляет реальные письма.

Пример запроса:

```text
Запусти LP-01 в тестовом режиме на данных из Сценарии/LP-01-daily-pmo-control/Входные данные. Запиши выходные XLSX в Результат и покажи calculated rows / written rows.
```

### Реальный режим

Реальный режим используется для ежедневной работы PMO на актуальных проектных данных.

В реальном режиме агент:

1. Использует реальные данные проектов, матрицу эскалаций, правила и историю ответов.
2. Использует нормативные пороги: 4 часа напоминание, 8 часов L2, 24 часа L3.
3. Для Red без плана митигации или критической угрозы программному дедлайну идёт по обычной матрице эскалации с ожиданием таймера.
4. Для L1/L2 может готовить уведомления к автоотправке, если есть разрешённый канал.
5. Для L3/L4 всегда ставит `pending_approval` и ждёт подтверждения руководителя стрима / человека.
6. Записывает выходные XLSX и отдельно сообщает, сколько строк рассчитано и сколько записано.

Пример запроса:

```text
Запусти LP-01 в реальном режиме на актуальных PMO-файлах. L1/L2 можно готовить к отправке, L3/L4 только после подтверждения руководителя стрима. Запиши dashboard, escalation check, outbox и loop log.
```

## Роли человека

| Роль | Участие |
|---|---|
| РП / ответственный | Отвечает на запрос статуса, даёт комментарий, веху, дедлайн и план митигации. |
| Координатор / куратор блока | Получает L2, если РП не ответил или проблема требует координации. |
| Руководитель стрима | Утверждает L3/L4, принимает решение о мерах при аномалиях. |
| Руководитель программы | Получает L4 при критичных или неразрешённых отклонениях. |

Если аномалий нет, руководитель стрима подтверждает отсутствие эскалаций или принимает решение о дальнейших мерах.

## Правила статусов

`SK-01-project-status` нормализует статус только в `Green`, `Yellow` или `Red`.

| RAG | Когда используется |
|---|---|
| Green | Работа идёт по плану, блокеров нет, статус свежий, дедлайн не под угрозой. |
| Yellow | Есть риск, неполные данные, устаревший статус, близкий дедлайн или проблема с планом митигации. |
| Red | Дедлайн сегодня/прошёл, работа не завершена, есть критичный блокер, нет плана митигации или под угрозой программная поставка. |

Аномалиями считаются:

- RAG не Green;
- нет ответа 4 часа;
- статус старше 24 часов;
- дедлайн меньше 5 рабочих дней и статус не Green;
- дата окончания прошла, а работа не завершена;
- Red без плана митигации;
- нет ответственного;
- текст статуса противоречит датам или фактам.

## Правила эскалаций

`SK-06-escalation` маршрутизирует аномалии по уровням:

| Уровень | Адресат |
|---|---|
| L1 | РП / руководитель проекта |
| L2 | Координатор / куратор блока |
| L3 | Руководитель стрима |
| L4 | Руководитель программы |

Нормативные пороги реального режима:

| Условие | Действие |
|---|---|
| Нет ответа 4 часа | Напоминание L1 |
| Нет ответа 8 часов | Эскалация L2 |
| Нет ответа 24 часа | Эскалация L3 |
| Red без плана митигации | Эскалация по обычной матрице и таймеру |
| Критическая угроза программному дедлайну | Эскалация по обычной матрице и таймеру с повышенной срочностью в тексте |
| L4 без подтверждения больше 2 часов | Дублирующий канал |

L3 и L4 нельзя отправлять автоматически: они всегда требуют подтверждения человека.

## Входные файлы

Папка тестовых данных: `Сценарии/LP-01-daily-pmo-control/Входные данные/`

В реальном режиме можно использовать другую папку с такими же схемами.

### `pmo_projects.xlsx`

Лист: `Проекты`

| Поле | Смысл |
|---|---|
| Пункт | Идентификатор контролируемой строки / вехи |
| Номер проекта | Управленческий номер проекта |
| Название проекта | Название проекта |
| Название вехи | Контролируемая работа |
| Дата начала работ | Плановый старт |
| Дата окончания работ | Плановый дедлайн |
| Ответственный ФИО | РП или ответственный |
| Дата статуса | Дата получения текущего статуса |
| Сырой статус | Фактический текст статуса |
| План митигации | Наличие или описание плана снятия отклонения |
| Завершено | Признак завершения |

### `pmo_escalation_matrix.xlsx`

Лист: `Матрица`

| Поле | Смысл |
|---|---|
| Пункт | Ключ строки из `pmo_projects.xlsx` |
| Номер проекта | Номер проекта |
| Название проекта | Название проекта |
| Уровень | L1/L2/L3/L4 |
| ФИО | Адресат уровня |
| Email | Email для отправки или регистрации письма |
| Роль | Роль адресата |
| Канал | Канал коммуникации |

### `pmo_escalation_rules.xlsx`

Лист: `Правила`

| Поле | Смысл |
|---|---|
| Текущий уровень | `Нет`, `L1`, `L2` или `L3` |
| Следующий уровень | `L1`, `L2`, `L3` или `L4` |
| Если нет ответа минут | Порог ожидания. В live-режиме соответствует реальным минутам, в тесте может быть сжатой симуляцией. |
| Требуется подтверждение перед отправкой | `Да` для L3/L4 |
| Автоотправка разрешена | `Да` только для L1/L2 |
| Описание правила | Пояснение для PMO |

### `pmo_escalation_responses.xlsx`

Лист: `Ответы`

| Поле | Смысл |
|---|---|
| Пункт | Ключ строки из `pmo_projects.xlsx` |
| Номер проекта | Номер проекта |
| Название проекта | Название проекта |
| Уровень эскалации | Уровень, на котором пришёл ответ |
| Ответ после минут | Через сколько минут пришёл ответ |
| Дата ответа | Дата и время ответа |
| От кого | Автор ответа |
| Текст ответа | Содержание ответа |

## Выходные файлы

Папка по умолчанию: `Сценарии/LP-01-daily-pmo-control/Результат/`

Итоговый результат сценария хранится в одном файле:

### `pmo_agent_dashboard.xlsx`

Workbook содержит четыре листа.

### `Dashboard`

Сводный PMO-dashboard по проектам.

Поля: `Пункт`, `Номер проекта`, `Название проекта`, `RAG`, `Веха`, `Дедлайн`, `Ответственный`, `Аномалия`, `Рекомендуемое действие`.

### `Escalation_Check`

Проверка рассчитанных цепочек эскалации и точек остановки.

Поля: `Пункт`, `Номер проекта`, `Название проекта`, `RAG`, `Фактическая цепочка`, `Ответ получен`, `Уровень ответа`, `Остановка`, `Основание`.

### `Outbox`

Строки сообщений / черновиков для регистрации и отправки.

Поля: `Пункт`, `Номер проекта`, `Название проекта`, `Дата письма`, `Кому ФИО`, `Кому email`, `Уровень эскалации`, `Тема письма`, `Текст письма`, `Ответ получен`, `Требуется подтверждение`, `Статус отправки`.

`Текст письма` должен содержать проект, веху, проблему, RAG, дедлайн, ожидаемое действие и срок ответа.

Возможные значения `Статус отправки`:

- `draft_not_sent` - черновик, письмо не отправлено;
- `sent_allowed` - L1/L2 можно отправлять при наличии разрешённого live-канала;
- `pending_approval` - требуется подтверждение человека, обязательно для L3/L4.

### `Loop_Log`

Журнал запуска сценария.

Поля: `loop_id`, `business_date`, `checkpoint`, `started_at`, `active_projects_count`, `red_count`, `yellow_count`, `green_count`, `calculated_outbox_rows`, `written_outbox_rows`, `status`.

## KPI и контроль качества

| KPI | Норма |
|---|---|
| Покрытие статусами | >= 95% к 14:00 |
| Покрытие SK-01 | > 95% к 12:00 |
| Актуальность статусов | Обновление меньше 24 часов |
| Полный loop | Не больше 5 часов |
| Эскалация аномалий | 100% аномалий переданы в SK-06 |
| Адресация эскалаций | 100% адресатов взяты из матрицы |
| Скорость уведомления | Меньше 15 минут после триггера |

## Что агент должен сообщить в конце

В финальном ответе агент должен указать:

- режим работы: тестовый или реальный;
- какие входные таблицы использованы;
- сколько строк рассчитано;
- сколько строк фактически записано в XLSX;
- какие RAG и аномалии найдены;
- какие цепочки эскалаций построены;
- какие цепочки остановились из-за ответа;
- какие L3/L4 ожидают подтверждения;
- что реальные письма не отправлялись в тестовом режиме.

Нельзя писать, что файл обновлён, если фактической записи не было.
